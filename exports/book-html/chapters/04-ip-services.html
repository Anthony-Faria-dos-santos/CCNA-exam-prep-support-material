5  IP Services

5.1 Objectif du module

A l’issue de ce module, vous serez capable de :

  • Configurer et verifier le NAT inside source (statique, pool dynamique, PAT overload)
  • Configurer et verifier NTP en modes client et serveur, et interpreter la hierarchie stratum
  • Expliquer les roles de DHCP (processus DORA) et DNS dans un reseau d’entreprise
  • Decrire la fonction de SNMP (NMS, agent, MIB, GET/SET/TRAP/INFORM) et les differences entre v1/v2c/v3
  • Interpreter les facilities et les 8 niveaux de severite syslog
  • Configurer un pool DHCP serveur, un client DHCP et un relay (ip helper-address)
  • Expliquer le forwarding per-hop behavior (PHB) de la QoS : classification, marking, queuing, policing, shaping
  • Configurer l’acces distant securise via SSH (hostname, domain, cle RSA, VTY)
  • Decrire les capacites et usages de TFTP et FTP pour la gestion des fichiers reseau

Le module 4 represente environ 10 % de l’examen CCNA 200-301. Ce sont les services « invisibles » qui font tourner un reseau : personne ne les remarque quand ils fonctionnent, mais tout s’arrete des qu’ils sont mal configures. Autrement dit, c’est la plomberie de l’infrastructure — et c’est rarement testee sur la theorie pure, mais bien sur la capacite a lire une configuration et a identifier ce qui cloche.

5.2 4.1 — NAT statique, dynamique et PAT

Exam topic 4.1 : Configure and verify — inside source NAT using static and pools Niveau : Configure/Verify

5.2.1 Contexte

Imaginez une entreprise de 500 postes en adressage prive 10.0.0.0/8. Quand un employe ouvre son navigateur sur www.cisco.com, son paquet porte en source l’adresse 10.1.1.50 — une adresse RFC 1918 que aucun routeur d’Internet ne route. Sans mecanisme de traduction, ce paquet est silencieusement jete des la premiere hop sur Internet. Le NAT (Network Address Translation) est l’intermediaire qui reecrit l’adresse source pour la rendre routable, puis remet l’adresse d’origine sur les paquets retour. C’est grace a lui qu’IPv4 a survecu 30 ans au-dela de son epuisement theorique.

5.2.2 Theorie

Le NAT est ne dans les annees 1990 en reponse a l’epuisement des adresses IPv4. Le principe : utiliser des adresses privees (RFC 1918) en interne et traduire en adresses publiques a la frontiere. Trois variantes existent, chacune repondant a un cas d’usage precis.

Les quatre termes NAT a maitriser

Toute la terminologie Cisco repose sur deux axes : inside/outside (de quel cote du routeur NAT se trouve l’hote) et local/global (de quel point de vue on observe l’adresse).

Terme Definition Exemple
Inside local IP de l’hote interne, vue depuis l’interieur 10.1.1.50 (le PC)
Inside global IP de l’hote interne, vue depuis l’exterieur 203.0.113.10 (apres NAT)
Outside local IP de l’hote externe, vue depuis l’interieur 8.8.8.8 (DNS Google)
Outside global IP de l’hote externe, vue depuis l’exterieur 8.8.8.8 (identique en general)

Moyen mnemotechnique : Inside = votre reseau, Local = vu d’en dedans, Global = vu d’en dehors. La majorite du temps, seule l’adresse inside est traduite — d’ou l’expression inside source NAT.

Les trois variantes de NAT

1. NAT statique (static) — un mapping permanent un-pour-un entre une adresse privee et une adresse publique. Utilise pour les serveurs exposes (web, mail).

2. NAT dynamique (pool) — un pool d’adresses publiques partage a la volee. Premier arrive, premier servi. Si le pool est vide, les nouvelles connexions echouent.

3. PAT (Port Address Translation, alias NAT overload) — tous les hotes partagent une seule adresse publique, differencies par leur port source. C’est ce que fait votre box Internet domestique. Jusqu’a ~65 000 translations simultanees par adresse publique.

Critere NAT statique NAT dynamique PAT (overload)
Mapping 1:1 permanent 1:1 temporaire N:1 (port-based)
Direction Bidirectionnelle Inside -> Outside Inside -> Outside
Adresses publiques 1 par hote Pool partage 1 pour tous
Cas d’usage Serveurs publics Groupes d’utilisateurs Acces Internet general
Connexion entrante Oui Non Non (sauf port forwarding)

5.2.3 Schema : comparaison des trois types de NAT

Legende : le NAT statique consomme une adresse publique par serveur. Le pool dynamique partage quelques adresses entre les hotes actifs. Le PAT multiplexe des milliers d’hotes derriere une seule adresse publique grace aux numeros de port.

5.2.4 Schema : traduction PAT en action

Legende : le routeur NAT maintient une table bidirectionnelle. A l’aller, il reecrit la source. Au retour, il reecrit la destination. Le serveur web ne voit jamais l’adresse privee.

5.2.5 Mise en pratique CLI

NAT statique pour un serveur web

R1
! Mapping fixe 10.1.1.100 <-> 203.0.113.10
R1(config)# ip nat inside source static 10.1.1.100 203.0.113.10
!
! Designer les interfaces
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip nat inside
R1(config-if)# exit
R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip nat outside

NAT dynamique avec pool

R1
! Definir le pool d'adresses publiques
R1(config)# ip nat pool POOL-PUBLIC 203.0.113.10 203.0.113.20 netmask 255.255.255.0
!
! ACL selectionnant les hotes eligibles
R1(config)# access-list 1 permit 10.1.1.0 0.0.0.255
!
! Lier l'ACL au pool
R1(config)# ip nat inside source list 1 pool POOL-PUBLIC

PAT (overload)

R1
! Methode 1 : PAT via l'adresse de l'interface outside
R1(config)# access-list 1 permit 10.1.1.0 0.0.0.255
R1(config)# ip nat inside source list 1 interface GigabitEthernet0/1 overload

! Methode 2 : PAT via un pool reduit a 1 adresse
R1(config)# ip nat pool PAT-POOL 203.0.113.10 203.0.113.10 netmask 255.255.255.0
R1(config)# ip nat inside source list 1 pool PAT-POOL overload

Le mot-cle overload est la seule difference syntaxique entre NAT dynamique et PAT — mais la difference fonctionnelle est immense.

Important🎯 Point exam — NAT
  • Piege classique : oublier ip nat inside ou ip nat outside sur les interfaces. Sans ces directives, la regle de translation est ignoree. C’est la cause numero 1 des NAT qui « ne fonctionnent pas ».
  • Le mot-cle overload transforme un pool en PAT. Sans lui, chaque hote consomme une adresse complete du pool.
  • L’examen teste souvent les 4 termes inside/outside local/global avec des scenarios d’identification. Retenez : inside = votre reseau, global = vu d’Internet.

5.2.6 Exercice 4.1 — NAT statique + PAT combines

Contexte : l’entreprise NetCorp dispose d’un reseau 172.16.0.0/16 et d’un bloc public 198.51.100.0/28 (14 adresses utilisables). Le serveur web interne (172.16.1.100) doit etre joignable depuis Internet sur 198.51.100.10. Tous les autres postes doivent sortir en PAT sur l’adresse de l’interface outside (198.51.100.1).

Consigne : ecrivez la configuration complete du routeur NAT-GW (Gi0/0 inside, Gi0/1 outside).

💡 Indice Vous avez besoin de deux regles : une statique pour le serveur web, une dynamique avec overload pour les autres postes. L’ACL doit matcher 172.16.0.0/16 (wildcard 0.0.255.255).
🔬 Solution 
NAT-GW
NAT-GW(config)# interface GigabitEthernet0/0
NAT-GW(config-if)# ip address 172.16.0.1 255.255.0.0
NAT-GW(config-if)# ip nat inside
NAT-GW(config-if)# no shutdown
NAT-GW(config-if)# exit
!
NAT-GW(config)# interface GigabitEthernet0/1
NAT-GW(config-if)# ip address 198.51.100.1 255.255.255.240
NAT-GW(config-if)# ip nat outside
NAT-GW(config-if)# no shutdown
NAT-GW(config-if)# exit
!
! NAT statique pour le serveur web
NAT-GW(config)# ip nat inside source static 172.16.1.100 198.51.100.10
!
! PAT pour tout le reste
NAT-GW(config)# access-list 1 permit 172.16.0.0 0.0.255.255
NAT-GW(config)# ip nat inside source list 1 interface GigabitEthernet0/1 overload
Le statique donne au serveur web une adresse publique fixe et permet les connexions entrantes. Le PAT permet a tous les autres hotes de partager 198.51.100.1. Le wildcard 0.0.255.255 correspond a un /16.

5.3 4.2 — NTP : synchronisation des horloges

Exam topic 4.2 : Configure and verify — NTP operating in a client and server mode Niveau : Configure/Verify

5.3.1 Contexte

Un incident de securite survient a 14h32. Vous ouvrez les logs : le routeur affiche 14h28, le switch 15h45, le pare-feu 14h31. Reconstituer la chronologie devient un casse-tete. NTP (Network Time Protocol) resout ce probleme en synchronisant toutes les horloges sur une reference commune, avec une precision a la milliseconde. Sans NTP, les logs sont inexploitables, les certificats TLS peuvent etre rejetes, Kerberos echoue, et la correlation d’evenements est impossible.

5.3.2 Theorie

NTP est un protocole applicatif sur UDP port 123, concu en 1985 par David Mills. Il repose sur un modele hierarchique organise en strates (stratum). Plus le stratum est bas, plus la source est fiable.

5.3.3 Schema : hierarchie NTP stratum

Legende : le stratum 0 n’est jamais accessible directement par NTP (c’est le materiel de reference). Chaque niveau ajoute 1 au stratum. Au-dela de stratum 15, l’horloge est consideree comme non fiable. Stratum 16 = non synchronise.

Pourquoi NTP est critique

  • Correlation des logs : sans horloge synchronisee, impossible de retracer un incident
  • Certificats TLS/SSL : un certificat n’est valide que dans un intervalle temporel precis
  • Authentification Kerberos : tolerance de 5 minutes maximum entre client et serveur
  • Journalisation de conformite : ISO 27001, PCI-DSS exigent des logs dates et synchronises

5.3.4 Mise en pratique CLI

Configurer un routeur comme client NTP

R1
! Pointer vers un serveur NTP (interne ou public)
R1(config)# ntp server 10.1.1.254

! Optionnel : second serveur pour la redondance
R1(config)# ntp server 10.1.1.253

! Configurer le fuseau horaire
R1(config)# clock timezone CET 1
R1(config)# clock summer-time CEST recurring

Configurer un routeur comme serveur NTP master

R1
! Le routeur devient source NTP (a n'utiliser qu'en lab ou reseau isole)
R1(config)# ntp master 4
Warning⚠️ Attention — ntp master

ntp master fait generer le temps par le routeur lui-meme — a reserver aux labs ou aux reseaux sans acces a une source externe. En production, preferez un vrai serveur NTP (pool.ntp.org, serveur interne, appliance GPS).

Verification

R1
R1# show ntp status

nominal freq is 250.0000 Hz, actual freq is 249.9998 Hz, precision is 2**18
reference time is E4A1B2C3.4D5E6F70 (14:32:15.302 CET Fri Apr 4 2026)
clock offset is 0.5234 msec, root delay is 12.45 msec
root dispersion is 25.67 msec, peer dispersion is 1.23 msec

Interpretation : Clock is synchronized confirme la synchro reussie. Le stratum 5 signifie que R1 est a 5 sauts de l’horloge de reference. Un offset de 0.5 ms est excellent (< 1 ms).

R1
R1# show ntp associations

*~10.1.1.254      10.1.1.1         4     32     64   377  12.4    0.52    1.23

Important🎯 Point exam — NTP
  • Distinguez ntp server <ip> (le routeur est client) et ntp master <stratum> (le routeur devient source de temps). Piege classique a l’examen.
  • Le stratum d’un client est toujours stratum du serveur + 1.
  • NTP = UDP port 123. La synchronisation initiale peut prendre 5 a 15 minutes — ne paniquez pas si le premier show ntp status affiche unsynchronized.
  • Un routeur configure avec ntp server agit automatiquement comme serveur pour les equipements qui le pointent — pas besoin de ntp master dans ce cas.

5.3.5 Exercice 4.2 — Chaine de synchronisation

Contexte : chez MediaPlus, le serveur NTP central SRV-NTP (10.10.10.254, stratum 2) doit etre la reference. Le routeur R1 doit se synchroniser dessus, et les switches SW1 et SW2 doivent se synchroniser sur R1.

Consigne : ecrivez les commandes pour R1, SW1 et SW2. Indiquez le stratum final de chaque equipement.

💡 Indice R1 est a la fois client (de SRV-NTP) et serveur (pour les switches). Pas besoin de ntp master : des que R1 est synchronise, il agit automatiquement comme source pour ses clients.
🔬 Solution 
R1
R1(config)# ntp server 10.10.10.254
SW1
SW1(config)# ntp server 10.10.10.1
SW2
SW2(config)# ntp server 10.10.10.1
Stratums finals : SRV-NTP = 2, R1 = 3 (2+1), SW1 et SW2 = 4 (3+1).

5.4 4.3 — DHCP et DNS : roles dans le reseau

Exam topic 4.3 : Explain — the role of DHCP and DNS within the network Niveau : Explain

5.4.1 Contexte

Sans DHCP, chaque poste d’un reseau de 500 machines devrait etre configure a la main : IP, masque, passerelle, DNS. Une faute de frappe, et c’est le conflit d’adresses. Sans DNS, pour aller sur google.com, il faudrait taper 142.250.74.46. Ces deux protocoles sont invisibles pour l’utilisateur final, mais absolument fondamentaux dans tout reseau moderne.

5.4.2 Theorie

DHCP — Dynamic Host Configuration Protocol

DHCP fonctionne en client/serveur sur les ports UDP 67 (serveur) et UDP 68 (client). Son role est d’attribuer automatiquement une configuration IP complete aux clients.

Le processus DORA

L’attribution d’une adresse suit quatre etapes, resumees par l’acronyme DORA : Discover, Offer, Request, Acknowledge.

5.4.3 Schema : processus DHCP DORA

Legende : le DISCOVER et le REQUEST sont en broadcast (dst 255.255.255.255). Le REQUEST est broadcast meme si le client connait le serveur, afin d’informer les autres serveurs que leur offre a ete refusee et qu’ils peuvent liberer l’adresse qu’ils avaient reservee.

Parametres fournis par DHCP
Parametre Obligatoire Exemple
Adresse IP Oui 10.1.1.50
Masque de sous-reseau Oui 255.255.255.0
Passerelle par defaut De facto 10.1.1.1
Serveur(s) DNS De facto 10.1.1.254, 8.8.8.8
Duree de bail Oui 86400 s (24 h)
Nom de domaine Optionnel corp.netcorp.local
Le bail DHCP

Le client ne possede pas l’adresse — il la loue. A 50 % du bail, il tente un renouvellement (REQUEST unicast). A 87,5 %, nouveau retry. Si le bail expire, le client perd son adresse et relance DORA.

DNS — Domain Name System

DNS traduit les noms de domaine (www.cisco.com) en adresses IP (72.163.4.185). Il fonctionne sur UDP port 53 pour les requetes standard et TCP port 53 pour les transferts de zone ou reponses volumineuses.

5.4.4 Schema : resolution DNS recursive

Legende : du point de vue du client, la resolution est une seule question et une seule reponse. En realite, le resolveur local remonte la hierarchie DNS (racine -> TLD -> autoritaire) pour trouver la reponse. Le resultat est ensuite mis en cache pour les requetes suivantes.

Types d’enregistrements DNS courants
Type Role Exemple
A Nom -> IPv4 www.cisco.com -> 72.163.4.185
AAAA Nom -> IPv6 www.cisco.com -> 2001:420:1101:1::185
CNAME Alias vers un autre nom mail.corp.com -> outlook.office365.com
MX Serveur de messagerie corp.com -> mail.corp.com (priorite 10)
PTR IP -> nom (reverse DNS) 72.163.4.185 -> www.cisco.com

Important🎯 Point exam — DHCP et DNS
  • Ports DHCP : serveur UDP 67, client UDP 68. Mnemonique : 67 < 68, le serveur vient en premier.
  • DHCP utilise UDP, pas TCP. DISCOVER et REQUEST sont des broadcasts (dst 255.255.255.255).
  • DNS : UDP 53 pour les requetes normales, TCP 53 pour les reponses > 512 octets et les transferts de zone.
  • L’ordre DORA est un grand classique : Discover -> Offer -> Request -> Acknowledge.

5.4.6 Exercice 4.3 — Chronologie complete d’un navigateur

Contexte : chez TechnoVert, un PC vient d’etre branche sur le VLAN 10 (10.10.10.0/24, passerelle 10.10.10.1, DHCP 10.10.10.254). L’utilisateur ouvre son navigateur et tape www.technovert.fr.

Consigne : decrivez chronologiquement les 6 premieres interactions reseau (apres branchement du cable).

💡 Indice D’abord DHCP (le PC n’a pas d’adresse), puis DNS (il ne connait pas l’IP du site), puis HTTP.
🔬 Solution
  1. DHCP Discover (broadcast UDP, src 0.0.0.0:68 dst 255.255.255.255:67) — trouver un serveur DHCP
  2. DHCP Offer — le serveur 10.10.10.254 propose 10.10.10.50/24, GW 10.10.10.1, DNS 10.10.10.254
  3. DHCP Request (broadcast) — le PC accepte l’offre
  4. DHCP Acknowledge — le serveur confirme, le PC a sa configuration IP
  5. DNS Query (UDP 53) vers 10.10.10.254 : « Quelle est l’IP de www.technovert.fr ? »
  6. DNS Reply — le serveur repond avec l’adresse (ex : 185.12.34.56)
  7. (bonus) TCP SYN vers 185.12.34.56:443 — ouverture de connexion HTTPS
Sans DHCP, pas de configuration IP. Sans DNS, pas de resolution de nom. DHCP doit donc se produire en premier, puis DNS, puis seulement HTTP.

5.5 4.4 — SNMP : supervision reseau

Exam topic 4.4 : Explain — the function of SNMP in network operations Niveau : Explain

5.5.1 Contexte

Vous gerez 200 equipements sur 5 sites. Comment savoir en temps reel si un lien sature, si un switch chauffe, si la memoire d’un routeur atteint sa limite ? Vous n’allez pas vous connecter en SSH a chaque machine toutes les 5 minutes. SNMP (Simple Network Management Protocol) centralise la collecte des metriques et peut vous alerter instantanement en cas de probleme.

5.5.2 Theorie

SNMP est un protocole applicatif dedie a la supervision et a la gestion d’equipements reseau. Il repose sur trois composants : le NMS (Network Management Station, le superviseur), les agents (logiciels embarques dans chaque equipement) et la MIB (Management Information Base, la base de variables supervisables).

5.5.3 Schema : architecture SNMP manager/agent

Legende : le NMS interroge les agents sur le port UDP 161 (polling periodique : GET, GET-NEXT, GET-BULK, SET). Les agents envoient des notifications asynchrones (TRAP, INFORM) vers le NMS sur UDP 162 quand un evenement survient.

Les operations SNMP

Operation Direction Description
GET NMS -> Agent Lire une variable specifique
GET-NEXT NMS -> Agent Lire la variable suivante dans la MIB
GET-BULK NMS -> Agent Lire plusieurs variables en une requete (v2c+)
SET NMS -> Agent Modifier une variable
TRAP Agent -> NMS Notification asynchrone non acquittee
INFORM Agent -> NMS Notification avec acquittement (v2c+)

La difference fondamentale TRAP vs INFORM : le trap est fire and forget. L’inform exige un acquittement ; sans reponse, l’agent retransmet.

Versions SNMP

Critere SNMPv1 SNMPv2c SNMPv3
Authentification Community (clair) Community (clair) User + password (MD5/SHA)
Chiffrement Non Non Oui (DES, AES)
GET-BULK Non Oui Oui
INFORM Non Oui Oui
Usage recommande Obsolete Labs seulement Production

Les community strings de v1/v2c sont des mots de passe en clair : - RO (read-only) — lecture seule. Defaut : public - RW (read-write) — lecture + ecriture. Defaut : private

Warning⚠️ Attention — community strings par defaut

En production, changez toujours les communities public et private. Laisser ces valeurs, c’est comme laisser le mot de passe admin sur un routeur — n’importe quel scanner automatique les essaiera.

5.6 4.5 — Syslog : facilities et severites

Exam topic 4.5 : Describe — the use of syslog features including facilities and levels Niveau : Describe

5.6.1 Contexte

Un routeur redemarre a 3 h du matin. Premiere question de l’astreinte : « que s’est-il passe juste avant ? ». Les messages syslog sont le journal de bord de l’equipement — chaque evenement significatif y est consigne, des changements d’etat d’interface aux erreurs critiques. Savoir lire et organiser ces messages est le B.A.-BA du troubleshooting reseau.

5.6.2 Theorie

Syslog est un standard (RFC 5424) qui permet aux equipements d’envoyer des messages de journal a un collecteur centralise (port UDP 514 par defaut). Chaque message contient trois informations :

  1. Facility : la source (quel composant IOS a genere le message)
  2. Severity : la gravite (0 a 7)
  3. Message : le contenu descriptif

Les 8 niveaux de severite

C’est le tableau le plus important de cette section — a connaitre par coeur.

Niveau Mot-cle Description Exemple
0 Emergency Systeme inutilisable Crash systeme
1 Alert Action immediate Temperature critique
2 Critical Condition critique Erreur memoire
3 Error Condition d’erreur Erreur d’interface
4 Warning Avertissement Seuil memoire atteint
5 Notification Normal mais significatif Interface up/down
6 Informational Message informatif Transaction config
7 Debugging Debogage Output de debug

Mnemonique : Every Awesome Cisco Engineer Will Need Ice cream Daily.

Tip💡 Astuce — logique de filtrage syslog

Plus le numero est bas, plus c’est grave. Configurer un niveau inclut tous les niveaux de severite inferieure ou egale. logging trap warnings (niveau 4) capture aussi les niveaux 0, 1, 2 et 3.

5.6.3 Schema : decision de filtrage par severite

Legende : chaque destination syslog (console, buffer, VTY, host) a son propre seuil. Un message dont la severite numerique est <= au seuil configure est logge. logging trap informational (6) capture les niveaux 0 a 6 et ignore le debugging (7).

Anatomie d’un message syslog

*Apr  4 14:32:15.302: %LINEPROTO-5-UPDOWN: Line protocol on Interface
  GigabitEthernet0/1, changed state to down

Decomposition : %LINEPROTO = facility, 5 = severity (Notification), UPDOWN = mnemonique, puis description.

Destinations syslog

Destination Commande Persistance Usage
Console logging console <niveau> Non Debogage interactif
Terminal VTY terminal monitor Non (session) Debogage distant
Buffer local logging buffered <taille> <niveau> Perdu au reboot show logging
Serveur syslog logging host <ip> Oui Archivage, conformite

5.7 4.6 — DHCP : client et relay

Exam topic 4.6 : Configure and verify — DHCP client and relay Niveau : Configure/Verify

5.7.1 Contexte

La section 4.3 a explique la theorie de DHCP (DORA, bail, parametres). Passons a la pratique. Deux scenarios reviennent sans cesse en entreprise : configurer une interface routeur comme client DHCP (tres classique sur l’interface WAN chez un FAI), et configurer un relay DHCP pour que les clients d’un VLAN distant atteignent un serveur DHCP dans un autre sous-reseau.

5.7.2 Theorie

Le probleme du broadcast DHCP

Le message DHCP Discover est un broadcast L3 (dst 255.255.255.255). Or, les routeurs ne transferent pas les broadcasts — c’est leur comportement par defaut, et c’est heureux (eviter les tempetes de broadcast). Consequence : si le serveur DHCP n’est pas dans le meme sous-reseau que le client, le Discover n’arrive jamais.

5.7.3 Schema : topologie DHCP relay inter-VLAN

Legende : sans ip helper-address, le Discover s’arrete a R1. Avec la commande, R1 convertit le broadcast en unicast vers le serveur DHCP et insere son adresse d’interface receptrice dans le champ GIADDR du paquet DHCP. Le serveur utilise ce GIADDR pour selectionner le bon pool d’adresses.

Ce que fait ip helper-address

  1. Intercepte le broadcast DHCP sur l’interface cote client
  2. Insere l’adresse de l’interface receptrice dans le champ GIADDR (Gateway IP Address)
  3. Envoie en unicast au serveur DHCP specifie
  4. Le serveur utilise GIADDR pour choisir le pool
  5. La reponse retourne au relay, qui la remet au client
Tip💡 Astuce — helper-address ne se limite pas a DHCP

Par defaut, ip helper-address relaie 8 services UDP : DHCP (67, 68), TFTP (69), DNS (53), TACACS (49), NetBIOS Name/Datagram (137, 138), Time (37). Pour restreindre, utilisez no ip forward-protocol udp <port>.

Configurer un routeur comme serveur DHCP

Un routeur Cisco IOS peut aussi faire serveur DHCP pour les petites agences :

R1
! Exclure les adresses reservees (routeur, serveurs fixes)
R1(config)# ip dhcp excluded-address 10.10.10.1 10.10.10.20

! Creer le pool
R1(config)# ip dhcp pool VLAN10-POOL
R1(dhcp-config)# network 10.10.10.0 255.255.255.0
R1(dhcp-config)# default-router 10.10.10.1
R1(dhcp-config)# dns-server 10.10.20.100 8.8.8.8
R1(dhcp-config)# domain-name netcorp.local
R1(dhcp-config)# lease 0 12 0
! lease = 0 jours, 12 heures, 0 minutes
Warning⚠️ Attention — emplacement de excluded-address

ip dhcp excluded-address est une commande globale, pas une commande du pool. Si vous la tapez en mode dhcp-config, IOS l’interprete comme un nom de pool et l’exclusion n’a aucun effet. Erreur tres frequente.

5.8 4.7 — QoS : classification, marking, queuing, policing, shaping

Exam topic 4.7 : Explain — the forwarding per-hop behavior (PHB) for QoS, such as classification, marking, queuing, congestion, policing, and shaping Niveau : Explain

5.8.1 Contexte

Sur un lien WAN, un appel VoIP et un telechargement de 2 Go se disputent la meme bande passante. Sans QoS, le telechargement noie la voix — la conversation devient hachee. La QoS (Quality of Service) permet de dire au reseau : « la voix passe en priorite, le telechargement attend ». C’est un arbitre qui garantit que les applications critiques obtiennent les ressources dont elles ont besoin.

5.8.2 Theorie

PHB (Per-Hop Behavior) signifie que chaque equipement prend ses propres decisions. Il n’y a pas de reservation de bout en bout (comme ATM le faisait). Chaque routeur applique ses politiques localement, en se basant sur le marquage des paquets.

5.8.3 Schema : pipeline QoS classification -> marking -> queuing

Legende : le pipeline suit l’ordre classification -> marking -> queuing -> scheduling. Policing (drop) et shaping (delay) agissent en parallele pour controler le debit. WRED est declenche quand une file approche de sa capacite maximale.

1. Classification

Identifier le type de trafic selon differents criteres :

Critere Couche Exemple
IP source/destination 3 ACL permit 10.1.1.0 0.0.0.255
Port TCP/UDP 4 Port 5060 (SIP), 16384-32767 (RTP)
Valeur DSCP 3 EF = 46
Valeur CoS (802.1Q) 2 CoS 5 pour la voix
NBAR (reconnaissance d’application) 7 YouTube, Teams, Zoom

2. Marking (marquage)

Apres classification, le paquet est « tampone » pour que les equipements suivants le traitent sans refaire l’analyse. Deux champs principaux :

  • CoS (Class of Service) — 3 bits dans le tag VLAN 802.1Q (0-7). Layer 2, perdu des que la trame quitte le VLAN.
  • DSCP (Differentiated Services Code Point) — 6 bits dans l’en-tete IP (0-63). Layer 3, preserve de bout en bout.
Classe DSCP binaire Decimal CoS Usage
Voice (EF) 101110 46 5 Voix RTP
Video (AF41) 100010 34 4 Videoconference
Critical Data (AF31) 011010 26 3 Apps metier
Transactional (AF21) 010010 18 2 Transactions BD
Best Effort 000000 0 0 Trafic par defaut

Notation AFxy : x = classe (1-4, 4 prioritaire), y = probabilite de drop (1 faible, 3 elevee). EF (Expedited Forwarding) = priorite absolue, latence minimale — reserve a la voix.

3. Queuing — CBWFQ et LLQ

CBWFQ (Class-Based Weighted Fair Queuing) : chaque classe a sa file avec une bande passante minimum garantie. LLQ (Low Latency Queue) : file prioritaire reservee a la voix et a la video temps reel — servie en premier, toujours.

4. Congestion avoidance — WRED

Quand une file approche de sa capacite, WRED (Weighted Random Early Detection) commence a rejeter aleatoirement des paquets avant saturation. Les paquets a probabilite de drop elevee (AF13) sont jetes avant les AF11. Cela evite la synchronisation TCP (tail drop) et lisse la chute de debit.

5. Policing vs Shaping

Critere Policing Shaping
Action si depassement Drop ou re-mark Delay (buffer)
Latence ajoutee Non Oui
Direction Entree ou sortie Sortie uniquement
Effet sur le debit Coupure nette Lissage progressif
Analogie Radar : exces de vitesse = amende Feu de circulation : regule le flux
Usage typique FAI limitant un client Adapter le debit au lien distant

5.9 4.8 — Acces distant securise via SSH

Exam topic 4.8 : Configure — network devices for remote access using SSH Niveau : Configure

5.9.1 Contexte

Se connecter a un routeur via Telnet, c’est comme envoyer un mot de passe sur une carte postale — n’importe qui sur le chemin peut le lire. SSH (Secure Shell) chiffre toute la session : authentification, commandes, outputs. C’est le standard absolu pour l’acces distant aux equipements reseau. Aucun reseau d’entreprise serieux n’utilise Telnet en 2026.

5.9.2 Theorie

SSH est un protocole applicatif sur TCP port 22. Deux versions existent :

  • SSHv1 : obsolete, vulnerable — a ne jamais utiliser
  • SSHv2 : version courante, AES, authentification par cle ou mot de passe

5.9.3 Schema : SSH vs Telnet

Legende : Telnet transmet les credentials et toutes les commandes en clair — un sniffer sur le chemin lit tout. SSH chiffre l’integralite de la session avec AES, rendant le sniffing inutile. En production, transport input ssh bloque definitivement Telnet sur les lignes VTY.

5.10 4.9 — TFTP et FTP : transfert de fichiers reseau

Exam topic 4.9 : Describe — the capabilities and functions of TFTP/FTP in the network Niveau : Describe

5.10.1 Contexte

Un routeur qui perd sa configuration, un switch a mettre a jour, un admin qui sauvegarde les configs de 50 equipements avant une maintenance : tous ces scenarios necessitent un protocole de transfert de fichiers. TFTP et FTP sont les deux protocoles historiques pour ces operations.

5.10.2 Theorie

TFTP — Trivial File Transfer Protocol

TFTP est volontairement minimaliste. Concu a une epoque ou les equipements avaient tres peu de memoire, il fait le strict necessaire : transferer un fichier, point.

Caracteristique Detail
Transport UDP port 69
Authentification Aucune
Navigation repertoires Non
Taille blocs 512 octets (par defaut)
Fiabilite Acquittement bloc par bloc (stop-and-wait)
Chiffrement Aucun
Usage typique Boot reseau PXE, sauvegarde/restauration IOS, transfert de configs

TFTP est comme un guichet automatique sans menu : vous devez savoir exactement ce que vous voulez. Pas de liste, pas de mot de passe, pas de repertoire.

FTP — File Transfer Protocol

FTP est bien plus complet mais plus lourd. Il offre une vraie gestion de fichiers.

Caracteristique Detail
Transport TCP port 21 (controle) + TCP port 20 (donnees)
Authentification Username + password (en clair)
Navigation repertoires Oui (ls, cd, pwd)
Taille blocs Variable, negociee
Fiabilite TCP (windowing)
Chiffrement Aucun en standard (SFTP/FTPS existent mais hors scope CCNA)
Usage typique Transferts volumineux (images IOS), gestion centralisee

Comparatif synthetique

Critere TFTP FTP
Protocole UDP 69 TCP 20, 21
Authentification Non Oui
Lister les fichiers Non Oui
Vitesse Lent (stop-and-wait) Rapide (windowing)
Complexite Tres simple Plus complexe (2 connexions)
Securite Nulle Faible (credentials en clair)
Cas d’usage Boot PXE, petits fichiers Images IOS, transferts reguliers

Operations courantes en reseau

  1. Sauvegarder la config : copier running-config vers un serveur TFTP/FTP
  2. Restaurer la config : copier depuis le serveur vers le routeur
  3. Mettre a jour l’IOS : copier une nouvelle image IOS vers la flash
  4. Sauvegarder l’IOS : copier l’image flash vers un serveur (backup avant upgrade)

## 4.1 --- NAT statique, dynamique et PAT {#sec-topic-4-1}

> **Exam topic 4.1** : *Configure and verify* --- inside source NAT using static and pools
> **Niveau** : Configure/Verify

### Contexte

Imaginez une entreprise de 500 postes en adressage prive 10.0.0.0/8. Quand un employe ouvre son navigateur sur `www.cisco.com`, son paquet porte en source l'adresse 10.1.1.50 --- une adresse RFC 1918 que **aucun routeur d'Internet** ne route. Sans mecanisme de traduction, ce paquet est silencieusement jete des la premiere hop sur Internet. Le NAT (Network Address Translation) est l'intermediaire qui reecrit l'adresse source pour la rendre routable, puis remet l'adresse d'origine sur les paquets retour. C'est grace a lui qu'IPv4 a survecu 30 ans au-dela de son epuisement theorique.

### Theorie

Le NAT est ne dans les annees 1990 en reponse a l'epuisement des adresses IPv4. Le principe : utiliser des adresses privees (RFC 1918) en interne et traduire en adresses publiques a la frontiere. Trois variantes existent, chacune repondant a un cas d'usage precis.

#### Les quatre termes NAT a maitriser

Toute la terminologie Cisco repose sur deux axes : **inside/outside** (de quel cote du routeur NAT se trouve l'hote) et **local/global** (de quel point de vue on observe l'adresse).

| Terme | Definition | Exemple |
|-------|-----------|---------|
| **Inside local** | IP de l'hote interne, vue depuis l'interieur | 10.1.1.50 (le PC) |
| **Inside global** | IP de l'hote interne, vue depuis l'exterieur | 203.0.113.10 (apres NAT) |
| **Outside local** | IP de l'hote externe, vue depuis l'interieur | 8.8.8.8 (DNS Google) |
| **Outside global** | IP de l'hote externe, vue depuis l'exterieur | 8.8.8.8 (identique en general) |

Moyen mnemotechnique : *Inside* = votre reseau, *Local* = vu d'en dedans, *Global* = vu d'en dehors. La majorite du temps, seule l'adresse **inside** est traduite --- d'ou l'expression *inside source NAT*.

#### Les trois variantes de NAT

**1. NAT statique (static)** --- un mapping permanent un-pour-un entre une adresse privee et une adresse publique. Utilise pour les serveurs exposes (web, mail).

**2. NAT dynamique (pool)** --- un pool d'adresses publiques partage a la volee. Premier arrive, premier servi. Si le pool est vide, les nouvelles connexions echouent.

**3. PAT (Port Address Translation, alias NAT overload)** --- tous les hotes partagent **une seule** adresse publique, differencies par leur port source. C'est ce que fait votre box Internet domestique. Jusqu'a ~65 000 translations simultanees par adresse publique.

| Critere | NAT statique | NAT dynamique | PAT (overload) |
|---------|-------------|---------------|----------------|
| Mapping | 1:1 permanent | 1:1 temporaire | N:1 (port-based) |
| Direction | Bidirectionnelle | Inside -> Outside | Inside -> Outside |
| Adresses publiques | 1 par hote | Pool partage | 1 pour tous |
| Cas d'usage | Serveurs publics | Groupes d'utilisateurs | Acces Internet general |
| Connexion entrante | Oui | Non | Non (sauf port forwarding) |

## 4.2 --- NTP : synchronisation des horloges {#sec-topic-4-2}

> **Exam topic 4.2** : *Configure and verify* --- NTP operating in a client and server mode
> **Niveau** : Configure/Verify

### Contexte

Un incident de securite survient a 14h32. Vous ouvrez les logs : le routeur affiche 14h28, le switch 15h45, le pare-feu 14h31. Reconstituer la chronologie devient un casse-tete. NTP (Network Time Protocol) resout ce probleme en synchronisant toutes les horloges sur une reference commune, avec une precision a la milliseconde. Sans NTP, les logs sont inexploitables, les certificats TLS peuvent etre rejetes, Kerberos echoue, et la correlation d'evenements est impossible.

### Theorie

NTP est un protocole applicatif sur UDP port 123, concu en 1985 par David Mills. Il repose sur un modele hierarchique organise en **strates** (stratum). Plus le stratum est bas, plus la source est fiable.

### Schema : hierarchie NTP stratum {#fig-m4-ntp-stratum}

```{mermaid}
    S1[Stratum 1<br/>Serveur NTP racine]
    S2[Stratum 2<br/>pool.ntp.org]
    S3[Stratum 3<br/>Routeur d'entreprise R1]
    S4[Stratum 4<br/>Switches SW1, SW2]
    S0 --> S1
    S1 --> S2
    S2 --> S3
    S3 --> S4
```

*Legende* : le stratum 0 n'est jamais accessible directement par NTP (c'est le materiel de reference). Chaque niveau ajoute 1 au stratum. Au-dela de stratum 15, l'horloge est consideree comme non fiable. Stratum 16 = non synchronise.

#### Pourquoi NTP est critique

- **Correlation des logs** : sans horloge synchronisee, impossible de retracer un incident
- **Certificats TLS/SSL** : un certificat n'est valide que dans un intervalle temporel precis
- **Authentification Kerberos** : tolerance de 5 minutes maximum entre client et serveur
- **Journalisation de conformite** : ISO 27001, PCI-DSS exigent des logs dates et synchronises

## 4.3 --- DHCP et DNS : roles dans le reseau {#sec-topic-4-3}

> **Exam topic 4.3** : *Explain* --- the role of DHCP and DNS within the network
> **Niveau** : Explain

### Contexte

Sans DHCP, chaque poste d'un reseau de 500 machines devrait etre configure a la main : IP, masque, passerelle, DNS. Une faute de frappe, et c'est le conflit d'adresses. Sans DNS, pour aller sur `google.com`, il faudrait taper `142.250.74.46`. Ces deux protocoles sont invisibles pour l'utilisateur final, mais absolument fondamentaux dans tout reseau moderne.

### Theorie

#### DHCP --- Dynamic Host Configuration Protocol

DHCP fonctionne en client/serveur sur les ports **UDP 67** (serveur) et **UDP 68** (client). Son role est d'attribuer automatiquement une configuration IP complete aux clients.

##### Le processus DORA

L'attribution d'une adresse suit quatre etapes, resumees par l'acronyme **DORA** : Discover, Offer, Request, Acknowledge.

### Schema : processus DHCP DORA {#fig-m4-dora}

```{mermaid}
%%| label: fig-m4-dora
%%| fig-cap: "Le processus DORA d'obtention d'un bail DHCP"
sequenceDiagram
    participant C as Client (0.0.0.0)
    participant S as Serveur DHCP (10.1.1.254)
    C->>S: 1. DISCOVER (broadcast 255.255.255.255)
    Note over C,S: src=0.0.0.0:68 dst=255.255.255.255:67
    S->>C: 2. OFFER "Je te propose 10.1.1.50/24"
    C->>S: 3. REQUEST (broadcast) "J'accepte 10.1.1.50"
    Note over C,S: Broadcast pour prevenir les autres serveurs
    S->>C: 4. ACKNOWLEDGE "Confirme. Bail = 24h"
    Note over C: Configuration IP complete
```

*Legende* : le DISCOVER et le REQUEST sont en broadcast (dst 255.255.255.255). Le REQUEST est broadcast meme si le client connait le serveur, afin d'informer les autres serveurs que leur offre a ete refusee et qu'ils peuvent liberer l'adresse qu'ils avaient reservee.

##### Parametres fournis par DHCP

| Parametre | Obligatoire | Exemple |
|-----------|------------|---------|
| Adresse IP | Oui | 10.1.1.50 |
| Masque de sous-reseau | Oui | 255.255.255.0 |
| Passerelle par defaut | De facto | 10.1.1.1 |
| Serveur(s) DNS | De facto | 10.1.1.254, 8.8.8.8 |
| Duree de bail | Oui | 86400 s (24 h) |
| Nom de domaine | Optionnel | corp.netcorp.local |

##### Le bail DHCP

Le client ne possede pas l'adresse --- il la **loue**. A 50 % du bail, il tente un renouvellement (REQUEST unicast). A 87,5 %, nouveau retry. Si le bail expire, le client perd son adresse et relance DORA.

#### DNS --- Domain Name System

DNS traduit les noms de domaine (`www.cisco.com`) en adresses IP (`72.163.4.185`). Il fonctionne sur **UDP port 53** pour les requetes standard et **TCP port 53** pour les transferts de zone ou reponses volumineuses.

### Schema : resolution DNS recursive {#fig-m4-dns-recursive}

```{mermaid}
%%| label: fig-m4-dns-recursive
%%| fig-cap: "Resolution recursive d'un nom de domaine depuis un client"
sequenceDiagram
    participant PC as Client
    participant R as Resolveur local (10.1.1.254)
    participant RT as Serveur racine (.)
    participant TLD as Serveur TLD (.com)
    participant AUTH as Serveur autoritaire (cisco.com)
    PC->>R: Qui est www.cisco.com ?
    R->>RT: Qui gere .com ?
    RT->>R: Les serveurs TLD .com sont a X.X.X.X
    R->>TLD: Qui gere cisco.com ?
    TLD->>R: Le serveur autoritaire est Y.Y.Y.Y
    R->>AUTH: Quelle est l'IP de www.cisco.com ?
    AUTH->>R: 72.163.4.185
    R->>PC: www.cisco.com = 72.163.4.185
```

*Legende* : du point de vue du client, la resolution est une seule question et une seule reponse. En realite, le resolveur local remonte la hierarchie DNS (racine -> TLD -> autoritaire) pour trouver la reponse. Le resultat est ensuite mis en cache pour les requetes suivantes.

##### Types d'enregistrements DNS courants

| Type | Role | Exemple |
|------|------|---------|
| **A** | Nom -> IPv4 | www.cisco.com -> 72.163.4.185 |
| **AAAA** | Nom -> IPv6 | www.cisco.com -> 2001:420:1101:1::185 |
| **CNAME** | Alias vers un autre nom | mail.corp.com -> outlook.office365.com |
| **MX** | Serveur de messagerie | corp.com -> mail.corp.com (priorite 10) |
| **PTR** | IP -> nom (reverse DNS) | 72.163.4.185 -> www.cisco.com |

## 🎯 Point exam --- DHCP et DNS

- **Ports DHCP** : serveur **UDP 67**, client **UDP 68**. Mnemonique : 67 < 68, le serveur vient en premier.
- DHCP utilise **UDP**, pas TCP. DISCOVER et REQUEST sont des broadcasts (dst 255.255.255.255).
- DNS : **UDP 53** pour les requetes normales, **TCP 53** pour les reponses > 512 octets et les transferts de zone.
- L'ordre DORA est un grand classique : Discover -> Offer -> Request -> Acknowledge.
:::

**Consigne** : decrivez chronologiquement les 6 premieres interactions reseau (apres branchement du cable).

<details><summary>💡 Indice</summary>
D'abord DHCP (le PC n'a pas d'adresse), puis DNS (il ne connait pas l'IP du site), puis HTTP.
</details>

<details><summary>🔬 Solution</summary>

1. **DHCP Discover** (broadcast UDP, src 0.0.0.0:68 dst 255.255.255.255:67) --- trouver un serveur DHCP
2. **DHCP Offer** --- le serveur 10.10.10.254 propose 10.10.10.50/24, GW 10.10.10.1, DNS 10.10.10.254
3. **DHCP Request** (broadcast) --- le PC accepte l'offre

5. **DNS Query** (UDP 53) vers 10.10.10.254 : « Quelle est l'IP de www.technovert.fr ? »

7. *(bonus)* **TCP SYN** vers 185.12.34.56:443 --- ouverture de connexion HTTPS

</details>

## 4.4 --- SNMP : supervision reseau {#sec-topic-4-4}

> **Exam topic 4.4** : *Explain* --- the function of SNMP in network operations
> **Niveau** : Explain

### Contexte

Vous gerez 200 equipements sur 5 sites. Comment savoir en temps reel si un lien sature, si un switch chauffe, si la memoire d'un routeur atteint sa limite ? Vous n'allez pas vous connecter en SSH a chaque machine toutes les 5 minutes. SNMP (Simple Network Management Protocol) centralise la collecte des metriques et peut vous alerter instantanement en cas de probleme.

### Theorie

SNMP est un protocole applicatif dedie a la supervision et a la gestion d'equipements reseau. Il repose sur trois composants : le **NMS** (Network Management Station, le superviseur), les **agents** (logiciels embarques dans chaque equipement) et la **MIB** (Management Information Base, la base de variables supervisables).

### Schema : architecture SNMP manager/agent {#fig-m4-snmp-flow}

```{mermaid}
%%| label: fig-m4-snmp-flow
%%| fig-cap: "Flux SNMP entre NMS et agents : polling vs traps"
sequenceDiagram
    participant NMS as NMS (SolarWinds)
    participant A1 as Agent R1
    participant A2 as Agent SW1
    NMS->>A1: GET sysUpTime (UDP 161)
    A1->>NMS: Response : 15 jours
    NMS->>A2: GET-BULK ifInOctets (UDP 161)
    A2->>NMS: Response : tableau de compteurs
    Note over A1: Interface Gi0/1 tombe
    A1->>NMS: TRAP linkDown Gi0/1 (UDP 162)
    Note over NMS: Alerte generee
```

*Legende* : le NMS interroge les agents sur le port UDP 161 (polling periodique : GET, GET-NEXT, GET-BULK, SET). Les agents envoient des notifications asynchrones (TRAP, INFORM) vers le NMS sur UDP 162 quand un evenement survient.

#### Les operations SNMP

| Operation | Direction | Description |
|-----------|-----------|-------------|
| **GET** | NMS -> Agent | Lire une variable specifique |
| **GET-NEXT** | NMS -> Agent | Lire la variable suivante dans la MIB |
| **GET-BULK** | NMS -> Agent | Lire plusieurs variables en une requete (v2c+) |
| **SET** | NMS -> Agent | Modifier une variable |
| **TRAP** | Agent -> NMS | Notification asynchrone non acquittee |
| **INFORM** | Agent -> NMS | Notification avec acquittement (v2c+) |

La difference fondamentale **TRAP vs INFORM** : le trap est *fire and forget*. L'inform exige un acquittement ; sans reponse, l'agent retransmet.

#### Versions SNMP

| Critere | SNMPv1 | SNMPv2c | SNMPv3 |
|---------|--------|---------|--------|
| Authentification | Community (clair) | Community (clair) | User + password (MD5/SHA) |
| Chiffrement | Non | Non | Oui (DES, AES) |
| GET-BULK | Non | Oui | Oui |
| INFORM | Non | Oui | Oui |
| Usage recommande | Obsolete | Labs seulement | Production |

Les **community strings** de v1/v2c sont des mots de passe en clair :
- **RO** (read-only) --- lecture seule. Defaut : `public`
- **RW** (read-write) --- lecture + ecriture. Defaut : `private`

::: {.callout-warning icon=false}
## ⚠️ Attention --- community strings par defaut

En production, changez **toujours** les communities `public` et `private`. Laisser ces valeurs, c'est comme laisser le mot de passe `admin` sur un routeur --- n'importe quel scanner automatique les essaiera.
:::

## 🎯 Point exam --- SNMP

- Ports : **UDP 161** (NMS -> agent pour GET/SET), **UDP 162** (agent -> NMS pour TRAP/INFORM). Piege classique d'inversion.

- La MIB est une base hierarchique d'OID. `sysName` = 1.3.6.1.2.1.1.5.
:::

## 4.5 --- Syslog : facilities et severites {#sec-topic-4-5}

> **Exam topic 4.5** : *Describe* --- the use of syslog features including facilities and levels
> **Niveau** : Describe

### Contexte

Un routeur redemarre a 3 h du matin. Premiere question de l'astreinte : « que s'est-il passe juste avant ? ». Les messages syslog sont le journal de bord de l'equipement --- chaque evenement significatif y est consigne, des changements d'etat d'interface aux erreurs critiques. Savoir lire et organiser ces messages est le B.A.-BA du troubleshooting reseau.

### Theorie

Syslog est un standard (RFC 5424) qui permet aux equipements d'envoyer des messages de journal a un collecteur centralise (port **UDP 514** par defaut). Chaque message contient trois informations :

1. **Facility** : la source (quel composant IOS a genere le message)
2. **Severity** : la gravite (0 a 7)
3. **Message** : le contenu descriptif

#### Les 8 niveaux de severite

C'est le tableau le plus important de cette section --- a connaitre par coeur.

| Niveau | Mot-cle | Description | Exemple |
|--------|---------|-------------|---------|
| **0** | Emergency | Systeme inutilisable | Crash systeme |
| **1** | Alert | Action immediate | Temperature critique |
| **2** | Critical | Condition critique | Erreur memoire |
| **3** | Error | Condition d'erreur | Erreur d'interface |
| **4** | Warning | Avertissement | Seuil memoire atteint |
| **5** | Notification | Normal mais significatif | Interface up/down |
| **6** | Informational | Message informatif | Transaction config |
| **7** | Debugging | Debogage | Output de `debug` |

**Mnemonique** : *Every Awesome Cisco Engineer Will Need Ice cream Daily*.

::: {.callout-tip icon=false}
## 💡 Astuce --- logique de filtrage syslog

Plus le numero est **bas**, plus c'est **grave**. Configurer un niveau inclut **tous les niveaux de severite inferieure ou egale**. `logging trap warnings` (niveau 4) capture aussi les niveaux 0, 1, 2 et 3.
:::

### Schema : decision de filtrage par severite {#fig-m4-syslog-severity}

```{mermaid}
%%| label: fig-m4-syslog-severity
%%| fig-cap: "Filtrage des messages syslog par niveau de severite"
flowchart TD
    M[Message syslog arrive<br/>severite N] --> C{N <= niveau<br/>configure ?}
    C -->|Oui| A[Message logge<br/>vers destination]
    C -->|Non| D[Message ignore]
    A --> B[Console, buffer,<br/>VTY, serveur syslog]
```

*Legende* : chaque destination syslog (console, buffer, VTY, host) a son propre seuil. Un message dont la severite numerique est <= au seuil configure est logge. `logging trap informational` (6) capture les niveaux 0 a 6 et ignore le debugging (7).

#### Anatomie d'un message syslog

```{.default}
*Apr  4 14:32:15.302: %LINEPROTO-5-UPDOWN: Line protocol on Interface
  GigabitEthernet0/1, changed state to down
```

Decomposition : `%LINEPROTO` = facility, `5` = severity (Notification), `UPDOWN` = mnemonique, puis description.

#### Destinations syslog

| Destination | Commande | Persistance | Usage |
|-------------|----------|-------------|-------|
| Console | `logging console <niveau>` | Non | Debogage interactif |
| Terminal VTY | `terminal monitor` | Non (session) | Debogage distant |
| Buffer local | `logging buffered <taille> <niveau>` | Perdu au reboot | `show logging` |
| Serveur syslog | `logging host <ip>` | Oui | Archivage, conformite |

```{.cisco filename="R1"}
  admin on vty0 (10.1.1.50)
```

**Interpretation** : `%LINK-3-UPDOWN` est de severite 3 (Error) --- la couche physique a change d'etat. `%LINEPROTO-5-UPDOWN` (severite 5) suit --- le protocole de ligne bascule. L'ordre LINK down puis LINEPROTO down revele la cascade couche 1 -> couche 2.

## 4.6 --- DHCP : client et relay {#sec-topic-4-6}

> **Exam topic 4.6** : *Configure and verify* --- DHCP client and relay
> **Niveau** : Configure/Verify

### Contexte

La section 4.3 a explique la theorie de DHCP (DORA, bail, parametres). Passons a la pratique. Deux scenarios reviennent sans cesse en entreprise : configurer une interface routeur comme **client DHCP** (tres classique sur l'interface WAN chez un FAI), et configurer un **relay DHCP** pour que les clients d'un VLAN distant atteignent un serveur DHCP dans un autre sous-reseau.

### Theorie

#### Le probleme du broadcast DHCP

Le message DHCP Discover est un broadcast L3 (dst 255.255.255.255). Or, les routeurs **ne transferent pas** les broadcasts --- c'est leur comportement par defaut, et c'est heureux (eviter les tempetes de broadcast). Consequence : si le serveur DHCP n'est pas dans le meme sous-reseau que le client, le Discover n'arrive jamais.

### Schema : topologie DHCP relay inter-VLAN {#fig-m4-dhcp-relay}

```{mermaid}
%%| label: fig-m4-dhcp-relay
%%| fig-cap: "Relay DHCP (ip helper-address) entre deux sous-reseaux"
graph LR
    PC1[PC1<br/>VLAN 10] -- Broadcast<br/>DHCP Discover --> R1
    PC2[PC2<br/>VLAN 10] -- Broadcast --> R1
    R1[R1 relay<br/>Gi0/0: 10.10.10.1<br/>Gi0/1: 10.10.20.1]
    R1 -- Unicast<br/>GIADDR=10.10.10.1 --> SRV[SRV-DHCP<br/>10.10.20.100]
    SRV -- Unicast reponse --> R1
    R1 -. Broadcast ou unicast<br/>reponse au client .-> PC1
```

*Legende* : sans `ip helper-address`, le Discover s'arrete a R1. Avec la commande, R1 convertit le broadcast en unicast vers le serveur DHCP et insere son adresse d'interface receptrice dans le champ **GIADDR** du paquet DHCP. Le serveur utilise ce GIADDR pour selectionner le bon pool d'adresses.

#### Ce que fait `ip helper-address`

1. Intercepte le broadcast DHCP sur l'interface cote client
2. Insere l'adresse de l'interface receptrice dans le champ **GIADDR** (Gateway IP Address)
3. Envoie en unicast au serveur DHCP specifie
4. Le serveur utilise GIADDR pour choisir le pool
5. La reponse retourne au relay, qui la remet au client

::: {.callout-tip icon=false}
## 💡 Astuce --- helper-address ne se limite pas a DHCP

Par defaut, `ip helper-address` relaie **8 services UDP** : DHCP (67, 68), TFTP (69), DNS (53), TACACS (49), NetBIOS Name/Datagram (137, 138), Time (37). Pour restreindre, utilisez `no ip forward-protocol udp <port>`.
:::

#### Configurer un routeur comme serveur DHCP

Un routeur Cisco IOS peut aussi faire serveur DHCP pour les petites agences :

```{.cisco filename="R1"}
! Exclure les adresses reservees (routeur, serveurs fixes)

! Creer le pool

R1(dhcp-config)# network 10.10.10.0 255.255.255.0

R1(dhcp-config)# dns-server 10.10.20.100 8.8.8.8
R1(dhcp-config)# domain-name netcorp.local
R1(dhcp-config)# lease 0 12 0

```

## ⚠️ Attention --- emplacement de excluded-address

`ip dhcp excluded-address` est une commande **globale**, pas une commande du pool. Si vous la tapez en mode `dhcp-config`, IOS l'interprete comme un nom de pool et l'exclusion n'a aucun effet. Erreur tres frequente.
:::

## 4.7 --- QoS : classification, marking, queuing, policing, shaping {#sec-topic-4-7}

> **Exam topic 4.7** : *Explain* --- the forwarding per-hop behavior (PHB) for QoS, such as classification, marking, queuing, congestion, policing, and shaping
> **Niveau** : Explain

### Contexte

Sur un lien WAN, un appel VoIP et un telechargement de 2 Go se disputent la meme bande passante. Sans QoS, le telechargement noie la voix --- la conversation devient hachee. La QoS (Quality of Service) permet de dire au reseau : « la voix passe en priorite, le telechargement attend ». C'est un arbitre qui garantit que les applications critiques obtiennent les ressources dont elles ont besoin.

### Theorie

**PHB** (Per-Hop Behavior) signifie que **chaque equipement prend ses propres decisions**. Il n'y a pas de reservation de bout en bout (comme ATM le faisait). Chaque routeur applique ses politiques localement, en se basant sur le marquage des paquets.

### Schema : pipeline QoS classification -> marking -> queuing {#fig-m4-qos-pipeline}

```{mermaid}
%%| label: fig-m4-qos-pipeline
%%| fig-cap: "Pipeline de traitement QoS sur un routeur"
flowchart TD
    IN[Paquet entrant] --> CL[Classification<br/>ACL, NBAR, DSCP, CoS]
    CL --> MK[Marking<br/>DSCP L3 ou CoS L2]
    MK --> QU[Queuing<br/>LLQ, CBWFQ, files par classe]
    QU --> CA{Congestion ?}
    CA -->|Non| SCH[Scheduling<br/>envoi par priorite]
    CA -->|Oui| WRED[WRED drop selectif<br/>AF13 > AF12 > AF11]
    WRED --> SCH
    SCH --> OUT[Interface de sortie]
    QU -.-> POL[Policing<br/>drop excedentaire]
    QU -.-> SHP[Shaping<br/>bufferise et lisse]
```

*Legende* : le pipeline suit l'ordre classification -> marking -> queuing -> scheduling. Policing (drop) et shaping (delay) agissent en parallele pour controler le debit. WRED est declenche quand une file approche de sa capacite maximale.

#### 1. Classification

Identifier le type de trafic selon differents criteres :

| Critere | Couche | Exemple |
|---------|--------|---------|
| IP source/destination | 3 | ACL permit 10.1.1.0 0.0.0.255 |
| Port TCP/UDP | 4 | Port 5060 (SIP), 16384-32767 (RTP) |
| Valeur DSCP | 3 | EF = 46 |
| Valeur CoS (802.1Q) | 2 | CoS 5 pour la voix |
| NBAR (reconnaissance d'application) | 7 | YouTube, Teams, Zoom |

#### 2. Marking (marquage)

Apres classification, le paquet est « tampone » pour que les equipements suivants le traitent sans refaire l'analyse. Deux champs principaux :

- **CoS** (Class of Service) --- 3 bits dans le tag VLAN 802.1Q (0-7). Layer 2, perdu des que la trame quitte le VLAN.
- **DSCP** (Differentiated Services Code Point) --- 6 bits dans l'en-tete IP (0-63). Layer 3, preserve de bout en bout.

| Classe | DSCP binaire | Decimal | CoS | Usage |
|--------|-------------|---------|-----|-------|
| Voice (EF) | 101110 | **46** | 5 | Voix RTP |
| Video (AF41) | 100010 | **34** | 4 | Videoconference |
| Critical Data (AF31) | 011010 | **26** | 3 | Apps metier |
| Transactional (AF21) | 010010 | **18** | 2 | Transactions BD |
| Best Effort | 000000 | **0** | 0 | Trafic par defaut |

Notation **AFxy** : x = classe (1-4, 4 prioritaire), y = probabilite de drop (1 faible, 3 elevee). **EF** (Expedited Forwarding) = priorite absolue, latence minimale --- reserve a la voix.

#### 3. Queuing --- CBWFQ et LLQ

**CBWFQ** (Class-Based Weighted Fair Queuing) : chaque classe a sa file avec une bande passante minimum garantie. **LLQ** (Low Latency Queue) : file prioritaire reservee a la voix et a la video temps reel --- servie en premier, toujours.

#### 4. Congestion avoidance --- WRED

Quand une file approche de sa capacite, **WRED** (Weighted Random Early Detection) commence a rejeter aleatoirement des paquets **avant** saturation. Les paquets a probabilite de drop elevee (AF13) sont jetes avant les AF11. Cela evite la synchronisation TCP (tail drop) et lisse la chute de debit.

#### 5. Policing vs Shaping

| Critere | Policing | Shaping |
|---------|----------|---------|
| Action si depassement | **Drop** ou re-mark | **Delay** (buffer) |
| Latence ajoutee | Non | Oui |
| Direction | Entree ou sortie | Sortie uniquement |
| Effet sur le debit | Coupure nette | Lissage progressif |
| Analogie | Radar : exces de vitesse = amende | Feu de circulation : regule le flux |
| Usage typique | FAI limitant un client | Adapter le debit au lien distant |

#### Verifier une policy-map sur une interface


::: {.callout-important icon=false}
## 🎯 Point exam --- QoS

- **Policing = drop**, **Shaping = delay**. L'examen pose souvent la question sous forme de scenario : « FAI limite un client » -> policing. « Adapter le debit sortant au lien WAN » -> shaping.
- DSCP EF (46) = voix. DSCP est L3 (IP), CoS est L2 (802.1Q). Le CoS disparait au routage, le DSCP traverse.
- Le concept PHB : chaque routeur decide independamment sur la base du marquage. Pas de signalisation de bout en bout.
- Ordre du pipeline : classification -> marking -> queuing -> scheduling.
:::

## 4.8 --- Acces distant securise via SSH {#sec-topic-4-8}

> **Exam topic 4.8** : *Configure* --- network devices for remote access using SSH
> **Niveau** : Configure

### Contexte

Se connecter a un routeur via Telnet, c'est comme envoyer un mot de passe sur une carte postale --- n'importe qui sur le chemin peut le lire. SSH (Secure Shell) chiffre toute la session : authentification, commandes, outputs. C'est le standard absolu pour l'acces distant aux equipements reseau. Aucun reseau d'entreprise serieux n'utilise Telnet en 2026.

### Theorie

SSH est un protocole applicatif sur **TCP port 22**. Deux versions existent :

- **SSHv1** : obsolete, vulnerable --- a ne jamais utiliser
- **SSHv2** : version courante, AES, authentification par cle ou mot de passe

### Schema : SSH vs Telnet {#fig-m4-ssh-vs-telnet}

```{mermaid}
%%| label: fig-m4-ssh-vs-telnet
%%| fig-cap: "Comparaison SSH et Telnet sur le meme canal reseau"
flowchart LR
    subgraph TELNET["Telnet TCP 23"]
        A[Admin] -- "admin/cisco<br/>texte clair" --> T[Routeur]
        S1[Sniffer] -. capture .-> A
    end
    subgraph SSH["SSH TCP 22"]
        B[Admin] -- "AES-256 chiffre" --> U[Routeur]
        S2[Sniffer] -. capture illisible .-> B
    end
```

*Legende* : Telnet transmet les credentials et toutes les commandes en clair --- un sniffer sur le chemin lit tout. SSH chiffre l'integralite de la session avec AES, rendant le sniffing inutile. En production, `transport input ssh` bloque definitivement Telnet sur les lignes VTY.

#### Prerequis SSH sur un equipement Cisco

| Prerequis | Commande | Pourquoi |
|-----------|----------|---------|
| Hostname | `hostname R1` | Fait partie du nom de la cle RSA |
| Nom de domaine | `ip domain-name corp.local` | Domaine + hostname = nom de cle |
| Cle RSA (>= 2048 bits) | `crypto key generate rsa modulus 2048` | Echange de cles asymetrique |
| Utilisateur local | `username admin secret P@ss2026!` | Authentification |

::: {.callout-warning icon=false}
## ⚠️ Attention --- pas de cle sans domaine

Sans hostname et nom de domaine, `crypto key generate rsa` echoue avec `% Please define a domain-name first`. C'est la source d'erreur numero 1 en lab SSH.
:::

## 4.9 --- TFTP et FTP : transfert de fichiers reseau {#sec-topic-4-9}

> **Exam topic 4.9** : *Describe* --- the capabilities and functions of TFTP/FTP in the network
> **Niveau** : Describe

### Contexte

Un routeur qui perd sa configuration, un switch a mettre a jour, un admin qui sauvegarde les configs de 50 equipements avant une maintenance : tous ces scenarios necessitent un protocole de transfert de fichiers. TFTP et FTP sont les deux protocoles historiques pour ces operations.

### Theorie

#### TFTP --- Trivial File Transfer Protocol

TFTP est volontairement minimaliste. Concu a une epoque ou les equipements avaient tres peu de memoire, il fait le strict necessaire : transferer un fichier, point.

| Caracteristique | Detail |
|----------------|--------|
| Transport | **UDP port 69** |
| Authentification | **Aucune** |
| Navigation repertoires | Non |
| Taille blocs | 512 octets (par defaut) |
| Fiabilite | Acquittement bloc par bloc (stop-and-wait) |
| Chiffrement | Aucun |
| Usage typique | Boot reseau PXE, sauvegarde/restauration IOS, transfert de configs |

TFTP est comme un guichet automatique sans menu : vous devez savoir exactement ce que vous voulez. Pas de liste, pas de mot de passe, pas de repertoire.

#### FTP --- File Transfer Protocol

FTP est bien plus complet mais plus lourd. Il offre une vraie gestion de fichiers.

| Caracteristique | Detail |
|----------------|--------|
| Transport | **TCP port 21** (controle) + **TCP port 20** (donnees) |
| Authentification | Username + password (en clair) |
| Navigation repertoires | Oui (ls, cd, pwd) |
| Taille blocs | Variable, negociee |
| Fiabilite | TCP (windowing) |
| Chiffrement | Aucun en standard (SFTP/FTPS existent mais hors scope CCNA) |
| Usage typique | Transferts volumineux (images IOS), gestion centralisee |

#### Comparatif synthetique

| Critere | TFTP | FTP |
|---------|------|-----|
| Protocole | UDP 69 | TCP 20, 21 |
| Authentification | Non | Oui |
| Lister les fichiers | Non | Oui |
| Vitesse | Lent (stop-and-wait) | Rapide (windowing) |
| Complexite | Tres simple | Plus complexe (2 connexions) |
| Securite | Nulle | Faible (credentials en clair) |
| Cas d'usage | Boot PXE, petits fichiers | Images IOS, transferts reguliers |

#### Operations courantes en reseau

1. **Sauvegarder la config** : copier running-config vers un serveur TFTP/FTP
2. **Restaurer la config** : copier depuis le serveur vers le routeur
3. **Mettre a jour l'IOS** : copier une nouvelle image IOS vers la flash
4. **Sauvegarder l'IOS** : copier l'image flash vers un serveur (backup avant upgrade)

::: {.callout-important icon=false}
## 🎯 Point exam --- TFTP et FTP

- **TFTP = UDP 69**. **FTP = TCP 21** (controle) + **TCP 20** (donnees). Piege classique d'inversion.
- TFTP n'a **aucune** authentification. FTP en a, mais les credentials passent en clair. Ni l'un ni l'autre n'est securise --- utiliser **SCP** (base sur SSH) pour la securite reelle.
- Syntaxe `copy <source> <destination>`. `copy running-config tftp:` = **sauvegarde**. `copy tftp: running-config` = **restauration**.
- TFTP est lent par conception (stop-and-wait, blocs 512 octets). FTP est rapide grace au windowing TCP.
:::

### Exercice 4.9 --- Choix protocole pour migration IOS

**Contexte** : chez LogiTrans, il faut mettre a jour l'IOS de 15 routeurs pendant une fenetre de maintenance de 2 heures. L'image fait 550 Mo. Le serveur de fichiers est a 10.1.1.200 sur un lien 100 Mbps dedie. Debits effectifs : TFTP ~500 Ko/s (stop-and-wait), FTP ~10 Mo/s.

**Consigne** : recommandez TFTP ou FTP et justifiez. Estimez le temps total dans chaque cas.

<details><summary>💡 Indice</summary>
Calculez le temps par routeur = 550 Mo / debit. Multipliez par 15.
</details>

<details><summary>🔬 Solution</summary>

| Protocole | Debit | Temps/routeur | Total 15 routeurs |
|-----------|-------|---------------|-------------------|
| TFTP | 500 Ko/s | 1 100 s ~ **18 min** | ~**4 h 30** |
| FTP | 10 Mo/s | 55 s ~ **1 min** | ~**15 min** |

**Recommandation** : **FTP** sans hesiter. Avec TFTP, la migration prendrait 4 h 30 et sortirait largement de la fenetre de 2 heures. Avec FTP, 15 minutes suffisent, laissant le temps pour les verifications post-upgrade. La difference vient du mecanisme stop-and-wait de TFTP (acquittement de chaque bloc de 512 octets) contre le windowing TCP de FTP.
</details>
